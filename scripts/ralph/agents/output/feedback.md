### Detailed Findings

- **playerIdentities** (schema line 242-290): Has `importSessionId: v.optional(v.id("importSessions"))` at line 279. The field is populated during batch import at `playerImport.ts` line 675. But there is NO `by_importSessionId` index -- only `by_name_dob`, `by_userId`, `by_email`, `by_playerType`.

- **guardianIdentities** (schema line 166-208): No `importSessionId` field at all. Has `createdFrom: v.optional(v.string())` but no session tracking. Indexes: `by_email`, `by_userId`, `by_phone`, `by_postcode`, `by_name`.

- **guardianPlayerLinks** (schema line 339-401): No `importSessionId` field. No way to trace back to the import session. Indexes: `by_guardian`, `by_player`, `by_guardian_and_player`, `by_guardian_and_status`, `by_status`.

- **orgPlayerEnrollments** (schema line 403-458): Has `importSessionId: v.optional(v.id("importSessions"))` at line 440. Populated during batch import at `playerImport.ts` line 1036. But NO `by_importSessionId` index.

- **sportPassports** (schema line 498-560): No `importSessionId` field. Created during import at `playerImport.ts` line 1057-1068 but with no session tracking. Indexes: `by_playerIdentityId`, `by_player_and_sport`, `by_player_and_org`, `by_organizationId`, etc.

### Import Traceability Gap

The `batchImportPlayersWithIdentity` mutation in `playerImport.ts` only sets `importSessionId` on 2 of the 6 tables:
- `playerIdentities` (line 675): `importSessionId: args.sessionId`

The other 4 tables (`guardianIdentities`, `guardianPlayerLinks`, `sportPassports`, `skillAssessments`) are created without any import session tracking.

## 4. Implementation Guidelines Per Story

### US-P2.4-001: checkUndoEligibility Query

**File:** `packages/backend/convex/models/importSessions.ts`

```typescript
export const checkUndoEligibility = query({
  args: {
    sessionId: v.id("importSessions"),
  },
  returns: v.union(
    v.object({
      eligible: v.literal(true),
      timeRemainingMs: v.number(),
      recordCounts: v.object({
        playerIdentities: v.number(),
        guardianIdentities: v.number(),
        guardianPlayerLinks: v.number(),
        orgPlayerEnrollments: v.number(),
        sportPassports: v.number(),
        skillAssessments: v.number(),
      }),
      totalRecords: v.number(),
    }),
    v.object({
      eligible: v.literal(false),
      reason: v.string(),
    })
  ),
  handler: async (ctx, args) => {
    // 1. Get session, verify status === "completed"
    // 2. Check time window: completedAt + UNDO_WINDOW_MS > Date.now()
    // 3. Check for dependent data: manual assessments (assessmentType !== "import")
    //    linked to players from this import
    // 4. Count records per table using by_importSessionId index
    // 5. Return eligibility with record counts
  },
});
```

**Key patterns:**
- Use `authComponent.getAuthUser(ctx)` for auth
- Use `.withIndex("by_importSessionId", q => q.eq("importSessionId", args.sessionId))` for all 6 tables
- For dependent data check: query `skillAssessments` by `playerIdentityId` where `assessmentType !== "import"` -- NOTE: this will require fetching player IDs first, then checking assessments. Use batch fetch pattern.
- Define `UNDO_WINDOW_MS = 24 * 60 * 60 * 1000` as a module-level constant

### US-P2.4-002: undoImport Mutation

**File:** `packages/backend/convex/models/importSessions.ts`

```typescript
export const undoImport = mutation({
  args: {
    sessionId: v.id("importSessions"),
    reason: v.string(),
  },
  returns: v.object({
    success: v.boolean(),
    deletedCounts: v.object({
      playerIdentities: v.number(),
      guardianIdentities: v.number(),
      guardianPlayerLinks: v.number(),
      orgPlayerEnrollments: v.number(),
      sportPassports: v.number(),
      skillAssessments: v.number(),
    }),
  }),
  handler: async (ctx, args) => {
    // 1. Auth check: authComponent.getAuthUser(ctx)
    // 2. Re-verify eligibility (time window + no dependent data)
    // 3. Delete in dependency order:
    //    a. skillAssessments by importSessionId
    //    b. sportPassports by importSessionId
    //    c. orgPlayerEnrollments by importSessionId
    //    d. guardianPlayerLinks by importSessionId
    //    e. guardianIdentities by importSessionId
    //    f. playerIdentities by importSessionId
    // 4. Update session: status -> "undone", set undoneAt/undoneBy/undoReason
    // 5. Return deleted counts
  },
});
```

**Critical patterns:**
- MUST re-verify eligibility inside the mutation (race condition: eligibility could change between query and mutation)
- Use `.withIndex("by_importSessionId")` for ALL table queries, NEVER `.filter()`
- Delete with `for...of` loop calling `ctx.db.delete(doc._id)` for each record
- Convex mutations are atomic: if any delete fails, the entire mutation rolls back

**Transaction safety:**
- Convex limit: 16,000 writes per mutation, 32,000 scans per mutation
- A 200-player import with full guardians/passports/benchmarks: ~3,800 records
- A 500-player import: ~9,500 records -- still within limits
- For imports > 1,000 players: consider chunked approach using `ctx.scheduler.runAfter(0, internal.models.importSessions.undoChunk, {...})` with session locked to `"undoing"` status

## 5. Security Patterns

- **Auth:** Use `authComponent.getAuthUser(ctx)` in all new backend functions
- **Role check:** Verify user is admin/owner of the organization. On frontend, use `authClient.organization.getActiveMember()` pattern from existing import page (line 370-394). On backend, verify membership via `ctx.db.query("member").withIndex(...)` or equivalent Better Auth check.
- **Audit trail:** Store `undoneBy` (user ID), `undoneAt` (timestamp), `undoReason` (text) on the import session
- **Rate limiting:** Consider preventing multiple concurrent undo operations on the same session (check status !== "undone" and status !== "undoing")
- **Organization scoping:** The `undoImport` mutation must verify the session belongs to the user's organization

## 7. Risks and Mitigations

| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| Undo deletes records that other features depend on | Low | `checkUndoEligibility` checks for non-import assessments. 24-hour window limits accumulation. |
| Race condition: two admins undo the same import simultaneously | Low | Re-verify status inside mutation. Convex OCC handles concurrent writes. |
| Schema migration needed for existing records (4 tables missing `importSessionId`) | Certain | Field is `v.optional()` -- no migration needed. Existing records simply won't be undoable, which is correct. |
| Guardian shared across imports -- undo one import deletes guardian needed by another | Medium | See "Critical Risk: Shared Identity Records" below. |

### Critical Risk: Shared Identity Records

**This is the most architecturally significant risk.** The identity tables (`playerIdentities`, `guardianIdentities`) are platform-level, not org-scoped. If two separate imports create enrollments for the same player (matched by name+DOB), the `playerIdentity` will have the `importSessionId` from the FIRST import that created it (`wasCreated: true`), but the second import will have reused it (`wasCreated: false`).

**Undo strategy for shared records:**
- `playerIdentities`: Only delete if `importSessionId === sessionId` (i.e., this import CREATED it). If the identity was reused, skip deletion.
- `guardianIdentities`: Same logic -- only delete if created by this import.
- `guardianPlayerLinks`: Delete if both the guardian and player were created by this import, OR if the link itself was created by this import.
- `orgPlayerEnrollments`: Always safe to delete by `importSessionId` (org-scoped).
- `sportPassports`: Safe to delete if created by this import (track via `importSessionId`).
- `skillAssessments`: Safe to delete if created by this import (track via `importSessionId`).

This means the record counts returned by `checkUndoEligibility` should reflect what WILL be deleted, not just what has the matching `importSessionId`. Records where the identity was reused from a previous import should be excluded from the delete set but their enrollments/passports/assessments should still be deleted.
