# Bug Fix: Voice Note Disambiguation — Draft Not Generated After Player Classification

**Issue:** #537
**Title:** UAT testing of voice notes for a player classification
**Branch:** `jkobrien/537`

---

## Problem

When a voice note contains a player reference that the AI cannot confidently resolve (status: `needs_disambiguation`), the coach is directed to a disambiguation page to manually select the correct player. After the coach selects the correct player entity, **no insight draft was ever generated** for that claim, causing the insight to be silently lost.

Additionally, when the AI mishears a player's name (e.g. transcribes "Connor" when the coach said "Cormac"), the AI-generated `title` and `description` on the draft would retain the incorrect name even after disambiguation.

---

## Root Cause

### Missing draft generation after disambiguation

The pipeline flow is:

1. `resolveEntities` action → stores entity resolutions → **immediately schedules `generateDrafts`**
2. `generateDrafts` runs → skips all `needs_disambiguation` claims (no resolved entity yet)
3. Coach visits the disambiguation page → selects correct player → `resolveEntity` mutation fires
4. `resolveEntity` updates `voiceNoteEntityResolutions` (status → `user_resolved`) and `voiceNoteClaims` (`resolvedPlayerName`, `resolvedPlayerIdentityId`)
5. ❌ **Nothing triggers `generateDrafts` again** — draft is never created — insight is lost

### Misheard name persists in draft text

When the AI mishears a player name (e.g. `rawText = "Connor"`, resolved to `"Cormac O'Brien"`), the claim `title` and `description` generated by the AI contain the misheard name. These were copied verbatim into the draft even after the coach corrected the entity via disambiguation.

---

## Fix

### 1. `packages/backend/convex/models/voiceNoteEntityResolutions.ts`

In the `resolveEntity` mutation (step 7.5, after the E6 batch update), schedule `generateDrafts` for the artifact:

```ts
await ctx.scheduler.runAfter(
  0,
  internal.actions.draftGeneration.generateDrafts,
  { artifactId: resolution.artifactId }
);
```

This ensures that after any disambiguation, draft generation re-runs for the whole artifact. The E6 batch may have resolved multiple claims with the same raw name in one call, so one `generateDrafts` run handles all of them.

### 2. `packages/backend/convex/actions/draftGeneration.ts`

**Idempotency:** Before the claim loop, fetch existing drafts for the artifact and build a Set of `claimId`s that already have a draft. Skip any claim in that set. This prevents duplicate drafts when `generateDrafts` is called a second time after disambiguation.

**Name correction:** Added `rawText` to the `ResolutionRecord` type and a `replaceEntityMention` helper. For `user_resolved` entities (coach disambiguation only), the helper replaces the raw mention text with the resolved player name in the draft's `title` and `description`, correcting misheard names in the AI-generated text.

## Files Modified

| File | Change |
|---|---|
| `packages/backend/convex/models/voiceNoteEntityResolutions.ts` | Schedule `generateDrafts` after `resolveEntity` completes (step 7.5) |
| `packages/backend/convex/actions/draftGeneration.ts` | Add `rawText` to `ResolutionRecord`; idempotency check; `replaceEntityMention` helper; apply name correction for `user_resolved` entities |

## Testing

1. Submit a voice note where the AI cannot confidently match the player name (`needs_disambiguation`)
2. Navigate to the disambiguation page and select the correct player
3. Return to the Drafts tab — the insight draft should now appear with the correct player name
4. Confirm the draft — the insight should apply to the player profile correctly

For the misheard name scenario:
1. Submit a voice note where the AI mishears the player name
2. Disambiguate to the correct player
3. The draft title and description should now reflect the correct player name, not the AI's mishearing
