## Codebase Patterns
**Last Updated**: 2026-02-28 - Iteration 1 (Phase 8)

### Architecture
- All routes scoped under `/orgs/[orgId]/`
- Turborepo monorepo: `apps/web/` (Next.js 14), `packages/backend/` (Convex)
- Better Auth for authentication, Convex for backend

### Convex Backend Patterns
- NEVER use `.filter()` — always use `.withIndex()`
- All functions need `args` and `returns` validators
- Use `Id<"tableName">` types, not `string`
- Index names include all fields: `by_orgId_and_status`
- Public queries/mutations via `api`, internal via `internal`
- Webhook handlers need `internalMutation`/`internalQuery` (no auth required)
- `"use node"` directive at top of file for Node.js actions (crypto, fetch, etc.)

### Quality Checks
- Run in order: `npx -w packages/backend convex codegen` then type check then biome check
- Biome needs multi-line braces: `if (x) {\n  return null;\n}` not `if (x) { return null; }`
- useTopLevelRegex: all regex must be at top-level scope
- noShadow: don't use `v` as a parameter name (shadows Convex `v` import)
- noIncrementDecrement: use `+= 1` not `++`
- useConsistentTypeDefinitions: use `type` not `interface`
- useBlockStatements: always use braces, multi-line format

### Phase 8 Specific Patterns
- Two separate API integrations: Twilio (existing) and Meta (new) -- NEVER mix
- WellnessDispatchService is the ONLY entry point for sending wellness checks
- WhatsApp Flows channel is STATELESS -- no whatsappWellnessSessions records
- wa_id (WhatsApp ID) must be immediately resolved to playerIdentityId -- never persist wa_id (GDPR)
- processFlowCompletionWebhook must always check idempotency before inserting health check

### File Locations (Phase 8)
- Meta actions: `packages/backend/convex/actions/metaWhatsapp.ts`
- Dispatch service: `packages/backend/convex/lib/wellnessDispatchService.ts`
- Session models: `packages/backend/convex/models/whatsappWellness.ts`
- Health check model: `packages/backend/convex/models/playerHealthChecks.ts`
- HTTP routes: `packages/backend/convex/http.ts`
